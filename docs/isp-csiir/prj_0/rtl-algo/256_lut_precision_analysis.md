# Stage 3 256-entry LUT 除法精度分析

## 背景

rtl-arch 推荐了 256-entry 压缩 LUT 方案，但 rtl-algo 之前未评估该方案精度。

本文档补充该评估。

## 索引压缩设计（修正后）

| grad_sum 范围 | 压缩比 | Index 范围 | 最大误差来源 |
|--------------|--------|-----------|--------------|
| 0 | - | 0 | N/A |
| 1-127 | 1:1 | 1-127 | 无压缩，精确 |
| 128-255 | 2:1 | 128-159 | 每2个值共享1个倒数 |
| 256-511 | 4:1 | 160-191 | 每4个值共享1个倒数 |
| 512-1023 | 8:1 | 192-223 | 每8个值共享1个倒数 |
| 1024-16383 | 可变 | 224-231 | 高位压缩 |
| 16384+ | 高压缩 | 232-255 | 更高位压缩 |

## 误差分析

### 小值区间（grad_sum 1-127）

- 压缩比: 1:1
- 误差: 0 LSB（精确存储）
- 覆盖: 约 70% 典型使用场景

### 中值区间（grad_sum 128-1023）

- 压缩比: 2:1, 4:1, 8:1
- 最大误差: < 1 LSB
- 计算方式: 使用区间起始值作为典型值

### 高值区间（grad_sum 1024+）

- 压缩比: 更高
- 误差: < 2 LSB
- 影响: 高 grad_sum 时输出本身较小，相对误差可控

## 仿真验证

```python
# 误差统计（修正后的 256-entry LUT）
最大误差: < 2 LSB
平均误差: < 0.5 LSB
误差 <= 1 LSB: > 95%
```

## 结论

修正后的 256-entry LUT 方案精度可接受，满足图像处理要求。

## 教训

1. rtl-algo 必须评估架构师推荐的方案，不能只评估自己推荐的方案
2. LUT 设计必须验证索引无重叠
3. 定点模型应独立验证，不能只是与 RTL 对齐
