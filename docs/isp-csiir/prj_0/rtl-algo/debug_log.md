## 问题记录

### 2026-03-22: 数据依赖分析修正

**问题描述**：
用户提出两个关键问题：
1. 行缓存行数是否合理？梯度计算需要 5x5 窗口，Stage 3 梯度融合需要 `grad(i, j+1)`（下一行梯度值）
2. 融合窗写回逻辑是否正确？伪代码将 blend_uv 写回窗口内所有行

**分析过程**：

#### 问题 1: Stage 3 前向依赖

分析算法原始定义：
```
grad_c = grad(i, j)       // 当前行
grad_u = grad(i, j-1)     // 上一行
grad_d = grad(i, j+1)     // 下一行 <-- 前向依赖！
```

**发现**：
- `grad_d` 需要下一行的梯度值
- 在流水线中，这需要特殊处理

**解决方案**：
- 方案 A: Stage 3 延迟 1 行处理，实时获取 grad_d
- 方案 B: 增加 1 行梯度缓存，预计算下一行梯度

**行缓存需求修正**：
- 像素行缓存: 4 行（足够）
- 梯度行缓存: 1-2 行（从 2 行优化）
- IIR 反馈缓存: 1 行（存储 avg_u）

#### 问题 2: 融合窗写回逻辑

分析伪代码：
```
for (h=-2; h<=2; h++)
    src_uv(i, j+h) = blend_uv(i, j)
```

**发现**：
- 这是软件/批处理模型的实现方式
- 在硬件流水线中不可行：
  - 写回 `src_uv(i, j+1)` 和 `src_uv(i, j+2)` 会覆盖未处理的像素
  - 违背因果性

**正确理解**：
- IIR 反馈通过水平混合实现
- 仅需存储上一行的 `avg0_u` / `avg1_u`
- 不需要复杂的多行写回

**硬件实现建议**：
- 简化 IIR：仅存储 1 行 avg_u 缓存
- 不执行复杂写回逻辑

**修正内容**：
- 更新 bitwidth_analysis.md 添加第 8 节"数据依赖分析"
- 明确前向依赖处理方案
- 修正行缓存需求
- 澄清 IIR 反馈机制

**影响范围**：
- rtl-arch/architecture.md 需要同步更新
- 行缓存设计需要调整
- Stage 3 时序需要重新规划

## 设计决策记录

### DD-001: 梯度行缓存简化
- **日期**: 2026-03-22
- **决策**: 梯度行缓存从 2 行简化为 1 行
- **理由**: 通过实时获取 grad_d，仅需存储上一行梯度
- **前提**: Stage 3 延迟 1 行处理

### DD-002: IIR 反馈简化
- **日期**: 2026-03-22
- **决策**: 不执行复杂的融合窗写回，仅存储 avg_u
- **理由**: 硬件流水线无法支持复杂写回逻辑
- **前提**: 简化算法不影响核心功能
